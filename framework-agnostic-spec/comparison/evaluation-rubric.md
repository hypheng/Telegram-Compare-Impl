# Framework Evaluation Rubric

## A. 框架能力对比

| 维度 | 关注点 | 证据 |
|---|---|---|
| UI 表达力 | 列表、输入框、复杂状态、动画 | 实际页面与代码 |
| 状态管理 | 单向数据流、异步状态、错误恢复 | 状态机与实现 |
| 网络与并发 | 请求模型、取消、重试、流式能力 | 数据层代码 |
| 存储 | 本地缓存、序列化、恢复 | 存储适配层 |
| 调试体验 | 日志、断点、堆栈、热重载 | 开发记录 |
| 测试能力 | 单测、集成测试、UI 测试 | 测试脚本 |
| 构建发布 | 环境搭建、构建速度、稳定性 | 构建日志 |

## B. AI 生成商用品能力对比

| 维度 | 关注点 | 记录方式 |
|---|---|---|
| Prompt 到代码 | 需要几轮提示才能可用 | 每切片记录 |
| 上下文稳定性 | AI 是否容易跑偏 | Review 记录 |
| 自动修复率 | 报错后 AI 能否自愈 | 修复日志 |
| 执行成本 | 单次任务耗时、人工介入、token 消耗 | AI delivery log |
| 代码可维护性 | 是否便于人类接手 | Review 结论 |
| 资料可得性 | 官方文档、社区样例、MCP 资料是否充分 | 调研结论 |
| 商业落地阻力 | 工具链、SDK、生态、许可 | TODO / 风险 |
| 问题沉淀质量 | 阻塞是否被结构化归因并进入 issue 管理 | parity matrix + issue tracker |

## 输出要求

每个功能切片结束后至少补齐:

- 完成状态
- 验收报告链接
- 主要阻塞项
- 人工 review 结论
- AI 协作观察
- 最新 AI delivery log 链接
- 相关 GitHub issue 链接
