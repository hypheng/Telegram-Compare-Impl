# S3 单聊详情与文本发送

## 切片目标

把 `S2` 的“进入聊天详情”从占位提示升级成真实可验收的单聊详情页，覆盖消息历史展示、文本输入、发送中、发送失败和失败重试，为后续 `S4 本地缓存与离线恢复` 提供稳定的消息域主路径。

## 用户价值

- 用户从会话列表点击任意会话后，能进入一个接近 Telegram 风格的聊天详情，而不是停留在占位页。
- 用户能浏览历史消息，输入文本并立即看到发送反馈。
- 用户在发送失败时，能看到明确失败状态并触发重试，而不是消息悄悄消失。

## 本切片范围

- 从 `S2 会话列表` 点击任意会话进入聊天详情。
- 展示会话标题、消息历史、消息时间和发送状态。
- 提供底部文本 composer 和发送按钮。
- 覆盖 `loading`、`ready`、`sending`、`failed`、`retrying` 五类主要状态。
- 允许通过 debug-only 控件强制下一条消息失败，以验证失败和重试路径。
- 返回会话列表后，最近消息预览与排序应反映刚完成的发送结果。

## 暂不纳入本切片

- 真实服务端推送、消息分页和已读回执。
- 图片、文件、语音、表情和附件能力。
- 群聊、多会话并发 composer 草稿恢复。
- 真正的在线状态、typing indicator 或复杂消息操作菜单。

## 交付假设

- 当前阶段继续允许 fixture-backed message repository。
- 发送中状态可由客户端先行表达，发送成功或失败由 demo repository 返回结果。
- 为了覆盖失败路径，demo build 可以保留 `下一条发送失败` 的 debug-only 控件；该控件不计入最终产品交互。
- 当前阶段如果真实 Figma 文件仍未就位，可继续用 repo-side handoff brief 代理 `D2`，但必须在 evidence 中明确缺口。

## 验收标准

| ID | 验收项 | 通过标准 |
|---|---|---|
| AC-S3-1 | 打开聊天详情 | 从列表点击会话后，先看到 detail loading，再进入聊天详情默认态 |
| AC-S3-2 | 消息历史可读 | 默认态至少展示会话标题、消息气泡、时间信息和输入区 |
| AC-S3-3 | 文本发送成功 | 输入非空文本后，消息先进入 sending，再转为 sent，并回写到会话列表预览 |
| AC-S3-4 | 发送失败可感知 | 强制失败时，消息气泡显式进入 failed 状态，并提供 retry 入口 |
| AC-S3-5 | 重试可恢复 | 点击 retry 后，失败消息可进入 retrying，再转为 sent |
| AC-S3-6 | 返回列表承接 | 从详情返回会话列表后，仍保留主壳上下文，且最近会话预览与排序反映最新发送 |

## 核心流程约束

### 打开聊天详情

1. 用户从 `S2` 列表点击会话行。
2. 详情页先进入 loading。
3. 数据可用时进入默认详情态。

### 发送文本

1. 用户输入非空文本并点击发送。
2. 页面立即表现 sending，而不是静默等待。
3. 发送成功后，消息变为 sent，composer 清空。

### 失败与重试

1. 当下一条发送被显式设置为失败时，消息进入 failed。
2. failed 消息保持在消息流中，并暴露 retry 入口。
3. retry 成功后，原失败消息变为 sent，不生成重复消息。

### 返回列表

1. 用户从详情返回列表。
2. 列表仍保留当前会话主壳，不回退到登录。
3. 最近消息预览、时间和会话排序反映刚完成的发送结果。

## 领域落点

- `ChatThread`: 会话摘要与消息流组合对象。
- `ChatDetailLoadResult`: `Success`、`Failed`。
- `SendMessageResult`: `Success`、`InvalidInput`、`Failed`。
- `RetryMessageResult`: `Success`、`Failed`。
- `ChatDetailRepository`: 详情读取、发送与失败重试边界。

## 对 KMP 的要求

- 共享层要明确 message 发送与重试语义，不把 Android 的 view/widget 状态直接塞进共享层。
- `androidApp` 必须把 `loading / ready / sending / failed / retrying` 显式映射到页面反馈。
- 详情页视觉应继续贴近 Telegram 风格，而不是回退成通用表单页。
