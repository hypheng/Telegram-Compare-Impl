# S1 登录与会话恢复

## 切片目标

用一个可本地验证的最小账号主路径，打通 Telegram 类应用的首次进入、已有会话恢复、失效会话回退登录三种启动场景，为后续 `S2 会话列表` 和 `S3 单聊详情与文本发送` 提供统一入口。

## 用户价值

- 用户打开应用后，能立即知道系统是在恢复会话还是需要重新登录。
- 用户在没有本地会话时，能完成一次最小登录并进入主壳。
- 用户在已保存会话失效时，能收到明确反馈并重新登录，而不是卡死在启动页。

## 本切片范围

- 应用冷启动时检查本地会话快照。
- 存在有效会话时直接恢复到主壳入口。
- 不存在会话时进入登录页。
- 会话快照失效时清空本地状态并给出可恢复错误提示。
- 提供手机号 + 验证码的 mock 登录链路。
- 登录成功后持久化会话，以便下次启动恢复。
- 提供登出动作，确保能回到登录页。

## 暂不纳入本切片

- 真实 Telegram API、短信验证码、风控或设备校验。
- 多账号切换。
- 完整 `S2` 会话列表能力，例如刷新、未读同步、搜索入口。
- 完整 `S4` 离线缓存策略。

## 交付假设

- 当前对比阶段允许使用 mock auth backend，固定 demo 验证码可写入实现文档和调试提示。
- 登录或恢复成功后的落点是 `Chat List Entry Shell`，用于证明导航到达主壳；`S2` 的完整列表验收仍单独计算。
- 为了验证恢复失败路径，demo build 可以包含一个显式调试入口来写入失效会话快照，但该入口不计入最终产品交互。

## 验收标准

| ID | 验收项 | 通过标准 |
|---|---|---|
| AC-S1-1 | 启动反馈 | 冷启动后 200ms 内可见 loading / restoring 反馈，不出现空白页 |
| AC-S1-2 | 无会话进入登录 | 本地没有会话时进入登录页，用户能理解下一步要输入什么 |
| AC-S1-3 | 失效会话回退 | 本地会话失效时显示明确错误，支持重试恢复或直接重新登录 |
| AC-S1-4 | 登录成功进入主壳 | 输入合法手机号和正确验证码后，进入 `Chat List Entry Shell` |
| AC-S1-5 | 会话可恢复 | 登录成功后冷启动应用，可自动恢复到同一用户会话 |
| AC-S1-6 | 登出清理状态 | 用户登出后清理本地会话，再次进入应用不应直接恢复 |

## 核心流程约束

### 首次进入

1. 应用启动。
2. 系统检查本地会话快照。
3. 未找到快照时进入登录页。
4. 用户输入手机号与验证码。
5. 登录成功后进入 `Chat List Entry Shell`。

### 恢复已有会话

1. 应用启动。
2. 系统检查本地会话快照。
3. 快照有效时显示恢复中反馈。
4. 恢复成功后直接进入 `Chat List Entry Shell`。

### 失效会话回退

1. 应用启动。
2. 系统检查本地会话快照。
3. 快照无效或过期时显示错误反馈。
4. 系统清理失效快照并跳转登录页。
5. 用户可重新登录。

## 领域落点

- `UserSession`: 当前登录用户、会话标识和展示信息。
- `SessionRestoreResult`: `Restored`、`NoSession`、`Failed`。
- `LoginResult`: `Success`、`InvalidInput`、`Failed`。
- `SessionRepository`: 恢复、登录、登出和持久化边界。

## 对双实现的要求

- KMP 与 CJMP 必须共享同一组启动路径语义，不能一边把“无会话”当错误，另一边当正常分支。
- 错误提示必须能追溯到明确状态，而不是散落在页面层里临时拼接。
- 会话恢复能力必须包含“成功、无会话、失效”三个结果，不能只实现 happy path。
