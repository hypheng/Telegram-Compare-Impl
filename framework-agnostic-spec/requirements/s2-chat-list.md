# S2 会话列表

## 切片目标

把 `S1` 的 `Chat List Entry Shell` 升级成真实可验收的 Telegram 风格会话列表，覆盖列表加载、默认展示、搜索、下拉刷新以及空态 / 错误态，为 `S3 单聊详情与文本发送` 提供稳定主壳。

## 用户价值

- 用户登录或恢复成功后，能立即看到熟悉的 Telegram 风格会话列表，而不是调试占位页。
- 用户能快速浏览会话标题、最近消息、时间和未读数，并通过搜索定位目标会话。
- 用户在刷新失败或列表为空时，能得到明确反馈并知道下一步操作。

## 本切片范围

- `S1` 成功落点切换到真实 chat list。
- 展示会话标题、最近消息预览、最近时间、未读数、静音状态和头像占位。
- 顶部区域包含中心标题、编辑入口、写消息入口和固定搜索框。
- 页面主壳必须撑满手机视口，顶部区域、搜索框、状态反馈区和底部导航保持固定。
- 提供本地搜索能力，支持按标题和最近消息预览过滤。
- 提供下拉刷新能力和非阻塞式刷新失败反馈。
- 覆盖 `loading`、`ready`、`empty`、`error`、`refreshing` 五类主要状态。
- 列表项点击后给出进入下一切片的承接反馈，但不纳入 `S3` 验收。

## 暂不纳入本切片

- 真实服务端分页、增量同步和多账号聚合。
- 会话置顶、滑动操作、批量编辑或归档。
- 搜索消息内容并直接定位到消息位置。
- 完整 `S3` 聊天详情页。

## 交付假设

- 当前对比阶段允许继续使用 fixture-backed chat repository。
- 下拉刷新可以通过 demo 数据更新时间、顺序或预览文案变化来证明工作，而不要求真实网络。
- 为了覆盖空态和错误态，demo build 可以保留显式调试入口切换 fixture scenario；该入口不计入最终产品交互。
- 默认 fixture 不能只有 3-4 条极简数据；至少要有足够多的会话让列表区域出现真实滚动，并覆盖长标题、长预览、静音、零未读和多未读等差异。
- 当前阶段如果真实 Figma 文件仍未就位，可先以 repo-side handoff brief 作为 KMP 实现代理，但必须把缺口回写到 evidence。

## 验收标准

| ID | 验收项 | 通过标准 |
|---|---|---|
| AC-S2-1 | 首次进入列表 | 登录成功或恢复成功后，能在 500ms 内看到列表 loading，并进入 chat list 默认态 |
| AC-S2-2 | 列表信息完整 | 默认态至少展示头像占位、会话标题、最近消息、时间和未读数 |
| AC-S2-3 | Telegram 风格层级 | 顶部结构、搜索框、列表密度和底部导航承载区符合 Telegram 风格约束，且页面主壳撑满视口 |
| AC-S2-4 | 搜索会话 | 输入关键词后列表能按标题或最近消息过滤；无结果时给出明确空态 |
| AC-S2-5 | 下拉刷新 | 用户可触发刷新，看到内联 refresh indicator；成功时列表内容更新，失败时给出非阻塞反馈；只有列表区域可滚动 |
| AC-S2-6 | 空态与错误态 | 无会话和加载失败都能展示可恢复状态，不出现空白页 |
| AC-S2-7 | 进入下一步承接 | 点击会话行有明确反馈，说明将进入聊天详情，但不误判为 `S3` 已完成 |

## 核心流程约束

### 默认进入列表

1. 用户从 `S1` 登录成功或恢复成功进入主壳。
2. chat list 先进入 loading。
3. 数据可用时进入默认列表态，页面固定 chrome 不因数据切换产生整页跳动。

### 搜索会话

1. 用户聚焦搜索框并输入关键词。
2. 列表按标题和最近消息预览过滤。
3. 命中结果时更新列表；无结果时进入搜索空态。

### 下拉刷新

1. 用户在列表顶部下拉。
2. 列表进入 refreshing。
3. 刷新成功时更新内容；失败时保持当前列表并展示可恢复反馈。
4. 刷新手势和滚动边界只发生在列表 viewport 内，不允许整页一起滚动。

### 空态与错误态

1. 当 fixture scenario 为 empty 时，展示空态文案和恢复入口。
2. 当 fixture scenario 为 error 时，展示错误态和重试入口。
3. 两种状态都不应阻塞登出和主壳导航。

## 领域落点

- `ChatSummary`: 会话列表摘要信息。
- `ChatListQuery`: 搜索关键词和过滤边界。
- `ChatListLoadResult`: `Success`、`Empty`、`Failed`。
- `ChatListRepository`: 获取列表、刷新列表的边界。

## 对 KMP 的要求

- 共享层先落列表语义、查询和 refresh 结果，不把 Android widget 行为塞进共享层。
- `androidApp` 必须把 `loading / ready / empty / error / refreshing / search-result` 显式映射到页面状态。
- 如果 `S2` 实现继续沿用原生 View，也要保持 Telegram 风格的密度和层级，而不是退回默认 Material 布局。
- demo fixture 必须足够丰富，能真实展示列表滚动、不同长度标题/预览和多种元信息组合。
